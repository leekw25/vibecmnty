# TR (Test Report)

## 1. 현재 테스트 상태
현재는 문서 기반 기획 단계로, 코드 베이스에서 실행 가능한 자동 테스트는 아직 없음.  
따라서 **현재 커버리지는 0% (아직 코드 부재 기준)**이다.

## 2. 테스트 철학
1. 기능 동작성보다 **권한·보안 시나리오를 먼저 보호**
2. CRUD만 통과했을 때 완성이라고 보지 않고, 예외 플로우(권한 없음/중복/중단 케이스)까지 포함
3. UI 동작은 자동화 테스트(E2E)로 핵심 사용자 경로만 고정
4. 회귀 위험이 큰 로직에 우선순위 부여

## 3. 테스트 범위(우선순위)

### P0 (출시 직전 필수)
- 회원가입/로그인/로그아웃
- 비인가 사용자 접근 차단
- 게시글 생성/수정/삭제
- 댓글/대댓글 저장 무결성
- 좋아요 중복 방지
- 투표 중복 제한/마감 처리
- 관리자 권한 페이지 진입 제어

### P1
- 조회수 집계 일관성
- 캘린더 일정 노출/정렬
- 이미지 업로드 정책(형식/크기 제한)
- 신고 처리 흐름

### P2
- 프로필 수정/표시
- 검색/필터 정렬 조합
- 통계 집계 화면

## 4. 테스트 전략
- Unit Test: `service` 계층 중심
- Integration Test: 인증 + 권한 + DB 트랜잭션
- E2E: 페이지 단위 주요 플로우
- 부하/성능: 초기에는 가벼운 스모크 테스트 후 사용자 증가 시 k6 또는 Artillery 도입

## 5. 보고 템플릿
- 실행 일시
- 실행 환경 (브라우저, Node, DB)
- 통과/실패 케이스 수
- 실패 원인 분류 (코드/데이터/환경)
- 다음 액션(재현 단계, 우선순위)
