# CBP (Coding Best Practices)

## 1. 코드 작성 원칙
1. **단순하고 명시적인 이름 사용**
   - `any` 사용 최소화, 도메인 의미가 드러나는 이름 사용
2. **단일 책임 함수**
   - 컨트롤러/액션은 입출력 처리, Service는 핵심 규칙 처리
3. **타입 안정성**
   - 모델/폼 검증 및 입력 폼(Form/ModelForm) 검증 규칙을 API 경계에 적용
4. **권한은 서버에서 강제**
   - UI 숨김은 참고용, 실제 차단은 백엔드 정책으로 처리
5. **실패의 명시성**
   - 에러는 조용한 실패 없이 사용자 메시지+로깅+코드별 분기

## 2. 폴더 및 파일 표준
- `app/` : 라우팅과 페이지(혹은 액션)
- `app/*/services/` (필요 시): 비즈니스 규칙 분리
- `app/*/models.py` : 데이터 정의
- `app/*/forms.py` : 입력 스키마(폼/유효성)
- `tests/` : 단위/통합 테스트
- `components/` : UI 컴포넌트
- `tests/` : 단위/통합 테스트

## 3. 네이밍 규칙
- DB 모델: PascalCase (Post, PostComment)
- 함수: camelCase
- 상수: SCREAMING_SNAKE_CASE 또는 UPPER_CAMEL_CASE
- 라우트 파일: 앱별 `urls.py`를 통해 기능 접두를 일관되게 구성

## 4. 보안 코딩 규약
- 비밀번호 저장 시 해시만 저장
- 세션 Cookie `HttpOnly`, `Secure`, `SameSite` 적용
- 관리자 동작은 감사 로그 기록
- 입력은 allowlist 기반 검증(특히 파일명, URL, HTML)
- SQL 인젝션은 ORM + 파라미터 바인딩으로 기본 차단

## 5. TDD 워크플로우
1. **RED:** 요구사항 기반 테스트 우선 작성
2. **GREEN:** 가장 단순한 구현으로 통과
3. **REFACTOR:** 중복 제거/도메인 책임 정리
4. `핵심도메인`(인증, 투표, 권한)은 반드시 실패 케이스 포함 테스트
5. E2E는 사용자 흐름마다 1개 이상 보강

## 6. 테스트 관행
- Commit 단위로 최소 1개 이상 테스트 갱신
- 마이그레이션 변경 시 스키마 회귀 테스트
- `@critical` 태그 기반 중요한 시나리오 우선 실행
- 커버리지 목표:
  - 전체: 70% 이상
  - 서비스 레이어: 90% 이상
- 민감 동작(투표/삭제/권한)은 Property-based 보조 테스트 고려

## 7. PR 리뷰 기준
- 권한 검사 빠짐 여부
- 검증 스키마 누락 여부
- 예외 처리 및 사용자 피드백 메시지
- 트랜잭션 경계가 필요한 로직 누락 여부
